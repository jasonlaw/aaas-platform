# Changelog

All notable changes to this platform setup are tracked here. The platform setup version is stored in `platform/VERSION`.

## Unreleased

## 0.15.0 - 2026-07-02

### Added
- **`platform/docker/Dockerfile` now bakes `faster-whisper` (speech-to-text) and `himalaya` (email CLI) into every tenant image.** Both are free, permissively licensed, and need no system-level (`apt`) dependencies: `faster-whisper` installs via `uv pip` into the tenant venv, `himalaya` installs as a self-contained binary to `/usr/local/bin` via its official installer script. Baked into the image rather than persisted at runtime, following the same precedent `hermes-agent` itself set for `hindsight-client` — "used constantly" capabilities belong in the image. This part alone is a tenant Docker image change only, no platform setup version implication (`docs/architecture.md#versioning`).
- **Tenant agents can now install additional runtime plugins (pip packages, standalone binaries) that survive `docker compose up --force-recreate`.** Previously, anything a tenant agent installed for itself at runtime landed in the container's writable layer and was silently discarded on the next recreate — the platform's standard response to tenant config edits, credential rotation, image upgrades, and troubleshooting recovery — with no error pointing at the cause (see `tenant-plugin-persistence-issue.md`). Fixed with three pieces, all copied into each tenant's volume at onboarding (`onboard-tenant.md` step 6.2.1) and backfilled for existing tenants (`upgrade-tenants.md`):
  - **`platform/docker/Dockerfile`** now sets `HERMES_DISABLE_LAZY_INSTALLS=0`, enabling the base image's own persisted-install target (`/opt/data/lazy-packages`, already on our mounted tenant volume but disabled by default upstream) and write-protects `/opt/hermes/.venv` from the `hermes` user (`chown root:root` + `chmod a-w`) — the venv was previously writable by the tenant agent, which upstream's own issue tracker documents as a way to crash the live gateway process, not just fail to persist (hermes-agent#7779).
  - **`platform/tenant-hermes/scripts/tenant-install.sh`** is the tenant agent's only supported way to install a pip package or binary at runtime: installs to `/opt/data/lazy-packages` or `/opt/data/.local/bin`, refuses anything targeting `/opt/hermes/`, and records the install in `/opt/data/installed-plugins.yaml`.
  - **`platform/tenant-hermes/scripts/reconcile-plugins.sh`** runs automatically on every container start (via the new **`tenant-entrypoint.sh`**, now the compose `command:` in place of the bare `gateway run`) and best-effort reinstalls anything the manifest lists that's missing or was built for a since-superseded Python interpreter; it never blocks container startup, only logs.
  - **`platform/tenant-hermes/SOUL.md.template`** now tells the tenant agent to use `tenant-install.sh` and never write into `/opt/hermes/`.
  - **`platform/sop/troubleshoot-tenant.md`** gains a "Tenant-Installed Plugin Missing Or Not Working" recovery path.
  - Tenant-specific/occasional-use plugins go through this mechanism; capabilities common enough to want for every tenant should instead be baked into the image (see the `faster-whisper`/`himalaya` entry above).

## 0.14.9 - 2026-07-02

### Fixed
- **Unattended watchdog escalations could reach a tenant `--force-recreate` with no operator present and no confirmation gate.** `aaas-watchdog.sh`'s `escalate()` runs `opencode run --auto`, which auto-approves every command with nobody able to intervene, and hands that session `troubleshoot-tenant.md` as the tenant incident playbook. That SOP's only check-in gate (step 11) explicitly disabled itself for `trigger: watchdog` runs ("no operator is present, so continue through the escalation path"), and none of its "Common Recovery Paths" (permission repair, invalid-config repair, knowledge-vault mount repair, Mnemosyne reactivation) had a recreate-specific guard of their own — so an unattended session could genuinely reach a full container replace on its own judgment. Same shape of exposure existed in `agent-vault-failure.md` Recovery B (re-provisions every active tenant and force-recreates each one), just less likely to trigger in practice since it also needs an operator-supplied API key. Fixed with a hard, explicit, no-exception rule rather than relying on SOP text an auto-approved session might reason around:
  - **`platform/scripts/aaas-watchdog.sh`**'s escalation prompt itself now states, in the text handed to the unattended OpenCode session, that it must never run `--force-recreate`/`down`/`rm` against any container for any reason in that session, and must instead apply only the non-recreate portion of a fix, then stop and write an alert/report naming the exact command the operator needs to run.
  - **`platform/sop/troubleshoot-tenant.md`** gains a "Container Recreate Policy" section up front (unattended: never recreate, no exception; attended: recreate always needs its own explicit operator confirmation, separate from any earlier confirmation) and every recovery path that recreates (Permission Denied, Invalid Config, Knowledge Vault mount repair, Mnemosyne reactivation) now says so explicitly at the point of the command. Step 11's unattended carve-out on the check-in gate is removed — unattended sessions now stop and alert instead of continuing to iterate when the evidence doesn't point to a narrow, non-recreate fix.
  - **`platform/incidents/agent-vault-failure.md`** Recovery B now states explicitly that unattended runs must stop before it (it needs a real operator-supplied API key and ends in recreating every active tenant, both forbidden unattended) and that attended runs must confirm before starting it and again before each tenant's recreate.
  - **`scripts/setup-platform.sh`**'s `validate_install()` gets matching regression guards so the watchdog prompt's restriction and `troubleshoot-tenant.md`'s policy section can't silently be removed later.
- **`update-tenant.md` and `manage-agent-vault.md` force-recreated without a confirmation step dedicated to the recreate itself.** Both SOPs already require the underlying change (a `.env`/`config.yaml` edit, a credential rotation) to be operator-confirmed, but the recreate that follows — a separate disruptive action, brief downtime, container replaced — had no confirmation of its own. `update-tenant.md` step 10 and `manage-agent-vault.md` section 2.5 (referenced by section 4's token revocation and `handle-tenant-request.md`'s `llm_key_change` flow) now require an explicit, separate y/n before the `--force-recreate` line, stating plainly what will happen and why.
- **`manage-agent-vault.md` section 2.5 pointed at a compose file that doesn't exist.** It used `-f /opt/aaas/tenants/{tenant-id}/docker-compose.yaml`, but every tenant service lives in the single shared `/opt/aaas/platform/docker/docker-compose.yaml` (per `onboard-tenant.md` step 8) — there is no per-tenant compose file anywhere in this platform. Corrected to the shared path (and added `--no-deps`, matching every other recreate command in the repo) so this command doesn't fail outright when actually run.

## 0.14.8 - 2026-07-02

### Fixed
- **Tenant Mnemosyne plugin activation could silently fail to persist across the very container recreate its own onboarding step ends with.** `onboard-tenant.md` step 12 pinned `HERMES_HOME=/opt/data` only on the first of three related commands (`mnemosyne-hermes install`), while the following `hermes config set memory.provider mnemosyne` and `hermes memory setup` ran with no `HERMES_HOME` override at all — and nothing in the tenant `.env`/compose service set one for normal runtime either, so which directory those two commands (and the long-running `gateway run` process itself) actually used was undefined. Since the step's last line is `docker compose up --force-recreate --no-deps -d`, any activation state that landed outside `/opt/data` would vanish at that exact moment. Fixed in three places:
  - **`platform/sop/onboard-tenant.md`** step 12 and **`docs/platform-setup.md`**'s copy of the same walkthrough now pin `-e HERMES_HOME=/opt/data` on all three `docker exec` commands, not just the install call. `docs/platform-setup.md` also had this sequence ending in a plain `docker compose restart` — changed to `--force-recreate --no-deps -d`, matching the platform-wide rule against `restart` for provider/config changes.
  - **`platform/tenant-hermes/env.template`** now sets `HERMES_HOME=/opt/data` for the tenant's normal runtime, so the long-running gateway process and the one-time activation commands are guaranteed to agree on which directory holds Mnemosyne's config/memory state.
  - **`scripts/setup-platform.sh`**'s `validate_install()` gets two new regression guards: all three activation commands in `onboard-tenant.md` step 12 must carry `-e HERMES_HOME=/opt/data`, and `tenant-hermes/env.template` must set `HERMES_HOME=/opt/data`.
  - No change to already-onboarded tenants automatically — this only fixes new onboardings and the documented recovery/re-run path (`troubleshoot-tenant.md`'s "Mnemosyne Not Active" recovery and `incidents/mnemosyne-seed-corruption.md` both point at the corrected step 12). Whether existing tenants need a one-time remediation pass is still open — see the "to discuss" item on documenting exactly what a tenant recreate preserves vs. discards.
- **The tenant harness only ever proved config *files* referenced Mnemosyne, never that the plugin was actually active in the running container.** `check-tenant.sh` checked `config.yaml` for `provider: mnemosyne` and `.env`/the container env for `MNEMOSYNE_DATA_DIR` — both static/config-level checks that would pass even if the activation state above never landed correctly, or was dropped by a recreate. Added a live check, `container_mnemosyne_active`, that runs `hermes memory status` (falling back to `hermes hermes-mnemosyne status`) inside the running container and looks for `mnemosyne` in the output; WARNs (not FAILs, since the container may legitimately be mid-restart) with a pointer to `mnemosyne-seed-corruption.md` if it doesn't. `scripts/setup-platform.sh`'s `validate_install()` gets a matching regression guard so this check can't silently be removed later.
- **`upgrade-tenants.md` force-recreated every active tenant on every run, even ones where the run's own backfill loop found nothing to change and the image was already current.** This directly conflicted with the platform's own rule against `docker compose down`/`restart` misuse for anything but a real need — recreating a container is not free (brief downtime, and anything living only in the container's writable layer is discarded) and was happening unconditionally regardless of whether it was actually necessary. Reworked step 3 to resolve the target image ID once up front, track a per-tenant `NEEDS_RECREATE` flag that backfill sub-steps set only when they actually change something for that tenant, and skip the recreate (recording `skipped_recreate: no changes` and still running verification/harness checks) when the running container's image already matches the target and nothing was backfilled. Also dropped the redundant `docker compose stop` + `docker compose rm -f` pair before `up --force-recreate` — the latter already stops and replaces the container in one step, so the extra pair only added downtime. Added a general rule to `PLATFORM-REFERENCE.md`: only `--force-recreate` a tenant container when something actually requires it, not as a reflexive default.

## 0.14.7 - 2026-07-02

### Fixed
- **`write-report.md` contradicted admin Hermes's own instructions on who may author a task report.** `write-report.md`'s "Trigger Field" section stated reports "are always written by OpenCode itself; Hermes (admin or tenant) is never the author" — but `admin-hermes/SOUL.md.template` already directs admin Hermes to write task reports itself, and admin Hermes (not OpenCode) is the agent that actually handles tenant-originated contact via `skills/handle-tenant-request.md`. Added a new `trigger: tenant_request` value, authored by admin Hermes, for when a tenant's `support_request`/`operator_alert`/`llm_key_change` is confirmed to be a valid platform-side issue. `skills/handle-tenant-request.md`'s `support_request` handling now explicitly requires writing this report once a complaint is confirmed valid, separately from (and in addition to) any knowledge-vault note for reusable patterns.

## 0.14.6 - 2026-07-01

### Added
- **`setup.sh`/`setup-platform.sh` gain a `--yes`/`--no-tty` flag.** `setup-platform.sh`'s same-version/version-mismatch confirmation prompt required `/dev/tty` and errored out in non-interactive contexts (automated/headless runs, CI, provisioning scripts) — reported previously in the 0.14.1 upgrade and worked around locally each time by patching in an auto-select. `--yes` (alias `--no-tty`) now makes `prompt_confirm_install()` auto-select "1. Continue with backup" without touching `/dev/tty` at all; `setup.sh` accepts the same flag and forwards it to `setup-platform.sh`. The no-TTY error path (for anyone who hits the prompt without the flag) now also names `--yes`/`--no-tty` as the fix.

### Fixed
- **The nohup fallback in `admin_hermes_restart()` was killed by systemd shortly after every restart, silently.** `aaas-watchdog.service` is `Type=oneshot`, run every 5 minutes by `aaas-watchdog.timer`, with no `KillMode=` override — so it defaulted to `KillMode=control-group`, which kills every process in the oneshot's cgroup when it exits. `nohup` only blocks `SIGHUP`; it does nothing against systemd's direct cgroup kill on service stop. So whenever the fallback path fired (now only the genuine "systemd `--user` unit not installed" case, after the fix above), the `nohup hermes dashboard --no-open &` it started was torn down along with the rest of the cgroup as soon as that tick's `aaas-watchdog.sh` returned — meaning the fallback never actually kept the process alive past the run that invoked it, and the next tick would find it dead again and retry the same doomed fallback. Added `KillMode=process` to the generated unit so systemd only kills the tracked main process (the script itself) on stop and leaves other cgroup members — i.e. the detached fallback child — running. Existing installs need `sudo ./aaas-watchdog.sh --install` re-run to regenerate the unit with this setting; it isn't picked up automatically.
- **Watchdog's escalation to OpenCode used a nonexistent CLI syntax, so it never actually ran the diagnosis/recovery prompt.** `aaas-watchdog.sh`'s `escalate()` invoked `opencode --non-interactive --workdir <dir> --message "<prompt>"` — none of `--non-interactive`, `--workdir`, or `--message` exist in the current `opencode` CLI (confirmed against `opencode run --help`), so every escalation printed opencode's help text and exited without doing anything, silently. Since restart-then-escalate is the last line of defense once a component won't come back on its own, this meant that path was dead for every entity (Agent Vault, any tenant, admin Hermes) since whichever release introduced the escalation flow. Changed to `opencode run --dir <dir> --auto "<prompt>"` — `run` is the actual non-interactive subcommand, `--dir` sets the working directory, `--auto` auto-approves permissions (deliberate: escalation only fires unattended, with no operator present to approve prompts), and the message is a positional argument rather than a flag value. Verified end-to-end against a simulated admin-Hermes failure: old syntax produced a help dump in the watchdog log, new syntax produced a completed autonomous diagnosis-and-recovery session with a written report.
- **Watchdog's admin-Hermes restart silently never used systemd, even when the `--user` unit was installed correctly.** `aaas-watchdog.sh` runs as a system unit (`aaas-watchdog.service`, `User=<operator>`) — a system unit with `User=` does not get an interactive login session, so `XDG_RUNTIME_DIR`/`DBUS_SESSION_BUS_ADDRESS` were never set and every `systemctl --user ...` call in `admin_hermes_restart()` failed to reach the session bus. That failure was masked by `&>/dev/null` on the guard check, so the function always fell through to the `nohup` fallback path — meaning the restarted process was never tracked by systemd and `Restart=on-failure` on `aaas-admin-hermes.service` protected nothing between watchdog ticks, regardless of install state. `aaas-watchdog.sh` now derives and exports `XDG_RUNTIME_DIR`/`DBUS_SESSION_BUS_ADDRESS` from its own UID at startup (also inherited by the `opencode` escalation subprocess, which hits the same `systemctl --user` calls via `hermes-admin-failure.md`), and the generated `aaas-watchdog.service` unit now sets matching `Environment=` lines directly (via `%U`) for anyone inspecting or copying the unit file on its own. The fallback log line no longer conflates "unit not installed" with "couldn't reach the session bus" — it now reports which one actually happened, including the underlying `systemctl` error, so this class of issue is diagnosable from the watchdog log alone next time.

## 0.14.5 - 2026-07-01

### Changed
- **Admin Hermes now has explicit persistence/troubleshooting guidance, mirroring the tenant agent's.** New "Persistence and Troubleshooting" section in `platform/admin-hermes/SOUL.md.template`: ask a clarifying question on unclear operator requests before acting; when no SOP/skill/playbook covers a problem, work it out using `PLATFORM-REFERENCE.md`, the relevant incident playbook, the knowledge vault, and actual system state before saying it can't be done (kept non-destructive while investigating — all existing confirmation/secret rules still apply once a fix is known); don't stop at the first failed attempt or escalate before checking the obvious downstream dependency; propose reliable new fixes as SOP improvements via `improve-sop.md` rather than quietly re-solving them each time; and report plainly, with exactly what's needed next, when genuinely blocked instead of giving up silently or claiming a task is done when it isn't. Existing tenants and already-provisioned admin installs pick this up on their next `SOUL.md` render (tenants via `upgrade-tenants.md`; admin Hermes has no automated re-render — the operator must manually diff and merge this section into `/opt/aaas/platform/admin/SOUL.md` on existing installs, same as any other `SOUL.md.template` change).
- **Admin Hermes no longer keeps a process log.** `aaas-admin-hermes.service` now sets `StandardOutput=null` / `StandardError=null` instead of appending to `hermes-admin.log`, so admin Hermes's stdout/stderr are discarded rather than written to a file or falling through to the systemd journal. Every manual/fallback restart path that used to write the same file (`aaas-watchdog.sh`'s `admin_hermes_restart()` nohup fallback, `skills/manage-agent-vault.md`'s credential-refresh restart, `incidents/hermes-admin-failure.md`'s Recovery A fallback) now discards output the same way instead. Diagnosis now relies entirely on process/service state and HTTP probes — `incidents/hermes-admin-failure.md`'s "Check Hermes admin log" diagnosis step is removed (steps renumbered), and `setup-admin-hermes.md` Step 3.1's Telegram-connectivity check no longer greps a log for "Connected to Telegram"; it instead treats the test message already sent in item 5 as the definitive signal, since that no longer has a log to fall back on.
- **Watchdog's own files moved into a dedicated `platform/watchdog/` folder, split into `logs/` and `state/`.** Previously `aaas-watchdog.sh` wrote its log to `platform/logs/aaas-watchdog.log` and its lock to `platform/state/aaas-watchdog.lock` — two directories that existed for, and were named after, no single component. Both now live under `platform/watchdog/` (`watchdog/logs/aaas-watchdog.log`, `watchdog/state/aaas-watchdog.lock`), which — combined with removing the admin-Hermes process log above — means `platform/logs/` and the old `platform/state/` no longer exist at all; the watchdog owns its whole folder outright. Updated every reference (`PLATFORM-REFERENCE.md`, `docs/architecture.md`, `incidents/hermes-admin-failure.md`) and added `watchdog/logs` and `watchdog/state` to the directories pre-created by `scripts/setup-prerequisites.sh` and `scripts/setup-platform.sh`.

### Added
- **Admin Hermes now installs as a systemd `--user` service** instead of a bare `nohup ... &`. New `platform/admin-hermes/aaas-admin-hermes.service`, installed by `setup-admin-hermes.md` Step 7 to `~/.config/systemd/user/`, gives it the same crash/reboot auto-restart guarantee (`Restart=on-failure`) that Docker's `restart: unless-stopped` already gives every other component. Requires `loginctl enable-linger <user>` (one-time, documented in Step 7) so it keeps running without an active login session on a headless server. The watchdog remains a second, independent layer on top of this — it checks whether the dashboard actually *responds*, not just whether the process is alive, and now restarts it via `systemctl --user restart aaas-admin-hermes.service` instead of its own `pkill`+`nohup` (falls back to the old path if the unit isn't installed yet, for instances that haven't re-run Step 7).

### Fixed
- **`aaas-watchdog.sh` was deployed non-executable:** the file is tracked in git at mode `100644`, and `setup-platform.sh`'s post-copy `chmod +x` block chmod'd thirteen other scripts individually but never included `aaas-watchdog.sh`, so a fresh install's `sudo aaas-watchdog.sh --install` failed with a permission error. Added the missing `chmod +x` line.
- **Watchdog lock file was unwritable by the operator it runs as:** `LOCK_FILE` was hardcoded to `/var/run/aaas-watchdog.lock`, but `/var/run` is root-owned while the 0.14.0 redesign moved the watchdog's systemd unit to run as `User=<operator>` (not root) — the operator's own `flock` open then failed silently on every run. Moved to `${PLATFORM_DIR}/state/aaas-watchdog.lock`, which is already operator-owned (see `setup-prerequisites.sh`'s `chown -R "$USER":"$USER" /opt/aaas`).
- **Watchdog and the recovery playbook both timed out before the dashboard finished its first-start build:** admin Hermes's dashboard does a TypeScript/Vite build on its first start after install/upgrade, which can take up to ~60 seconds before it responds. `aaas-watchdog.sh`'s shared `PROBE_TIMEOUT=15` and `hermes-admin-failure.md`'s Recovery A (16s wait) were both too short for this, causing false "restart failed" escalations to OpenCode while the dashboard was still building. Added a separate `ADMIN_HERMES_PROBE_TIMEOUT=90` used only for admin Hermes (`wait_until_healthy` now takes an optional per-call timeout override; Docker entities are unaffected and keep the 15s default, since their own HEALTHCHECK already absorbs startup time), and extended the incident playbook's wait to ~90s with a note explaining why.

## 0.14.4 - 2026-07-01

### Changed
- **Admin Hermes install redesigned to use the official Hermes per-user installer, and the dedicated `aaas` service account was removed entirely.** Previously `setup-admin-hermes.md` created a system account (`useradd --system --no-create-home --shell /usr/sbin/nologin aaas`), hand-built a venv at `/opt/aaas/admin/venv`, and `pip install`ed `hermes-agent[web,pty]` into it — a hand-rolled install path distinct from every other component in the platform, which is otherwise entirely owned by the operator account that ran `scripts/setup-prerequisites.sh` (which itself does `sudo chown -R "$USER":"$USER" /opt/aaas`). This produced two concrete problems: (1) every start/restart/watchdog action against admin Hermes had to shell out as `sudo -u aaas -H bash -c '...'`, since OpenCode and the rest of the platform run as `$USER` while the Hermes files were `aaas`-owned; (2) the skill's own "Known gap" note flagged Telegram support as broken via this pip path (`dashboard_auth` missing, `python-telegram-bot`/platform adapter not pulled in) and left it unconfirmed/unfixed. Root cause of (2): PyPI has historically lagged the `hermes-agent` git source (documented as serving 0.13.0 while source was at 0.14.0).
  - **`platform/skills/setup-admin-hermes.md`**: Preconditions no longer require `python3-venv` or a dedicated account; Step 1 now runs the official installer (`curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash -s -- --skip-browser`) as the operator, which clones and builds from git directly (sidestepping the PyPI lag that caused the Telegram gap), then adds `mnemosyne-memory[embeddings]` and `mnemosyne-hermes` into the installer's own managed venv via `uv pip install --python <discovered venv path>`; Step 2 no longer `chown`s the profile directory to `aaas` (it's already operator-owned); Step 7 drops the `sudo -u aaas -H bash -c` wrapper — commands run directly as the operator.
  - **`platform/scripts/aaas-watchdog.sh`**: the systemd unit installed by `--install` now runs as `User=<operator>` (captured once via `SUDO_USER`/`logname` at install time) instead of root, with `Environment=PATH=%h/.local/bin:...` — `%h` is resolved by systemd from `User=` at every run, so this is never a hardcoded path even if the unit file is copied elsewhere. `admin_hermes_restart()` drops its `sudo -u aaas` wrapper to match.
  - **`platform/skills/manage-agent-vault.md`** step 2's admin-credential-rotation restart command drops the same `sudo -u aaas` wrapper.
  - **`platform/scripts/eval-judge.sh`**: `ADMIN_HERMES` now resolves via `command -v hermes` (falling back to `$HOME/.local/bin/hermes`) instead of the now-nonexistent `/opt/aaas/admin/bin/hermes`.
  - **`README.md`** and **`docs/architecture.md`** updated to describe the new install location (`~/.local/bin/hermes`, `~/.hermes/hermes-agent/`, both outside `/opt/aaas/` and owned by the operator) in place of the retired `/opt/aaas/admin/` venv+binary split.
  - No change to `platform/admin-hermes/` (templates) or `platform/admin/` (rendered profile/secrets) — both are still git-tracked/deployed exactly as before; only the runtime that reads them changed.

## 0.14.3 - 2026-07-01

### Added
- **Optional fallback LLM provider, asked during tenant onboarding and admin Hermes setup.** Hermes Agent supports automatic cross-provider failover via a top-level `fallback_providers:` list in `config.yaml` (turn-scoped: switches to the backup provider:model if the primary fails on rate limits, server errors, or auth failures, then restores the primary on the next turn) — see https://hermes-agent.nousresearch.com/docs/user-guide/features/fallback-providers. Previously neither `onboard-tenant.md` nor `setup-admin-hermes.md` ever asked about this, so no tenant or admin instance had failover configured unless an operator hand-edited `config.yaml` afterward (and even then, the fallback API key would have had no path into Agent Vault, so nothing would register it with the Agent Vault proxy).
  - **`platform/sop/onboard-tenant.md`** step 1 now asks whether the operator wants a fallback provider (optional — declining is the common case). If given, step 5 renders a `fallback_providers:` entry into the tenant's `config.yaml`; step 6 verifies it (or verifies the block stayed commented out if declined); step 6.3 passes the fallback's provider var/key through to `provision-tenant-vault.md`; step 19's report includes fallback status.
  - **`platform/sop/provision-tenant-vault.md`** gains optional step 2.1 (store the fallback credential and register its service/hostname with Agent Vault, so the proxy will actually route to it) and step 4.1 (scrub the fallback key from `.env` to the `routed-via-agent-vault` placeholder, same as the primary), plus a matching check in step 6. Both credentials live in the same per-tenant vault. Skipped entirely if no fallback provider was collected.
  - **`platform/skills/setup-admin-hermes.md`** "Ask The Operator" gains item 1.1 for the admin agent's own optional fallback provider/model/key. New Step 5.7 stores the fallback credential in `admin-vault` and scrubs it from `admin/.env`, mirroring Steps 5.1–5.6 for the primary key. Step 3 and the Reporting section updated accordingly.
  - **`platform/tenant-hermes/config.yaml.template`** and **`platform/admin-hermes/config.yaml.template`** each gain a commented-out `fallback_providers:` example with a link to the upstream doc, left untouched if no fallback provider is configured.
  - **`platform/tenant-hermes/env.template`** and **`platform/admin-hermes/env.template`** each gain a commented placeholder line documenting that an optional fallback provider's API key follows the same `routed-via-agent-vault` placeholder pattern as the primary key, under its own env var name.
  - **`docs/platform-setup.md`** and **`docs/tenant-agent-setup.md`** updated to include the new optional questions in the illustrative onboarding walkthroughs.
