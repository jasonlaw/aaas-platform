# Plan B: Hermes Tenant Agent Setup
> **AaaS Platform — Tenant Agent Implementation Plan**
> Version: 1.2 (POC)
> Last Updated: 2026-06-15

## Overview

Hermes is the **tenant-facing agent** of the AaaS platform. Each restaurant owner
gets their own isolated Docker container running a Hermes instance, pre-configured
with their brand identity via Mnemosyne memory, and accessible via their own
Telegram bot.

This plan validates the full tenant experience end-to-end — from container startup
to content generation.

**Prerequisites:** Plan A (OpenCode Admin Agent Setup) fully completed and validated.

## Tenant Folder Structure Reference

```
/opt/aaas/tenants/{tenant-id}/      →  mounted as /opt/data inside container
├── config.yaml                         Hermes config (no secrets)
├── .env                                Secrets (API keys, tokens — never commit)
├── .env.template                       Secret keys reference (no values)
├── SOUL.md                             Agent personality
├── memories/
│   ├── MEMORY.md                       Brand seed (one-time Mnemosyne seed only)
│   └── USER.md                         Owner seed (one-time Mnemosyne seed only)
├── skills/                             Auto-generated by Hermes
├── sessions/                           Conversation history (Hermes managed)
├── logs/                               Runtime logs (Hermes managed)
└── files/                          →  mounted as /home/hermes/files inside container
    ├── assets/                         Logo, brand colors (seeded at onboarding)
    ├── uploads/                        Owner-sent files via Telegram
    └── generated/                      Agent-produced output (posters, captions)
```

**Volume mounts (official Hermes convention):**
```
HOST                                          CONTAINER
/opt/aaas/tenants/{id}/          →           /opt/data/
/opt/aaas/tenants/{id}/files/    →           /home/hermes/files/
```

## Phase 1: First Tenant — Manual Onboarding

Onboard the first tenant **manually** before trusting OpenCode to automate it.
This validates the full stack hands-on and is the reference for the SOP.

### 1.1 Create tenant config directory

```bash
TENANT_ID="test-restaurant"
```

```bash
mkdir -p /opt/aaas/tenants/$TENANT_ID/memories
mkdir -p /opt/aaas/tenants/$TENANT_ID/skills
mkdir -p /opt/aaas/tenants/$TENANT_ID/files/assets
mkdir -p /opt/aaas/tenants/$TENANT_ID/files/uploads
mkdir -p /opt/aaas/tenants/$TENANT_ID/files/generated
```

### 1.2 Write config.yaml

Create `/opt/aaas/tenants/test-restaurant/config.yaml`:

```yaml
_config_version: 1

model:
  provider: openai          # replace with your BYOK provider
  model: gpt-4o             # replace with your model

# Native Hermes memory DISABLED — Mnemosyne handles all memory
memory:
  memory_enabled: false
  user_profile_enabled: false

terminal:
  backend: local

display:
  tool_progress: off
  busy_ack_detail: off
  cleanup_progress: true

gateway:
  platforms:
    telegram:
      home_chat_id: ""      # auto-populated on first message
      gateway_restart_notification: true

# Mnemosyne plugin — path relative to /opt/data inside container
plugins:
  - name: mnemosyne
    enabled: true
    path: /opt/data/plugins/mnemosyne
```

### 1.3 Write .env file

Create `/opt/aaas/tenants/test-restaurant/.env`:

```bash
# Tenant secrets — DO NOT COMMIT THIS FILE
LLM_API_KEY=sk-your-test-key-here
TELEGRAM_BOT_TOKEN=your-bot-token-here
TELEGRAM_ALLOWED_USERS=your-numeric-telegram-user-id
```

**How to get your Telegram numeric user ID:**

```
1. Open Telegram
2. Search for @userinfobot
3. Send /start
4. Copy the numeric ID it replies with
5. Paste into TELEGRAM_ALLOWED_USERS
```

### 1.4 Write .env.template

Create `/opt/aaas/tenants/test-restaurant/.env.template`:

```bash
# Required secret keys for this tenant
# Copy to .env and fill in real values

LLM_API_KEY=
TELEGRAM_BOT_TOKEN=
TELEGRAM_ALLOWED_USERS=
# DISCORD_BOT_TOKEN=        # uncomment when Discord added
# WHATSAPP_BRIDGE_TOKEN=    # uncomment when WhatsApp added
```

## Phase 2: Add Tenant to Docker Compose

### 2.1 Add service to docker-compose.yaml

Add the following service to `/opt/aaas/platform/docker/docker-compose.yaml`:

```yaml
services:
  hermes_test-restaurant:
    image: hermes-tenant:latest
    container_name: hermes_test-restaurant
    restart: unless-stopped
    command: gateway run
    volumes:
      - /opt/aaas/tenants/test-restaurant:/opt/data
      - /opt/aaas/tenants/test-restaurant/files:/home/hermes/files
    env_file:
      - /opt/aaas/tenants/test-restaurant/.env
    deploy:
      resources:
        limits:
          memory: 1g
          cpus: "1.0"
```

### 2.2 Start tenant container

```bash
cd /opt/aaas/platform/docker
```

```bash
docker compose up -d hermes_test-restaurant
```

### 2.3 Verify container is running

```bash
docker ps | grep hermes_test-restaurant
```

### 2.4 Check startup logs

```bash
docker logs hermes_test-restaurant --tail 30
```

Look for:
- Mnemosyne plugin loaded
- Telegram gateway connected
- No errors

## Phase 3: Seed Mnemosyne Memory

### 3.1 Wait for container to fully initialise

```bash
sleep 10
```

### 3.2 Seed brand context into Mnemosyne

Seed brand facts:
```bash
docker exec hermes_test-restaurant hermes memory add "$(cat /opt/aaas/tenants/test-restaurant/memories/MEMORY.md)"
```

Seed owner profile:
```bash
docker exec hermes_test-restaurant hermes memory add "$(cat /opt/aaas/tenants/test-restaurant/memories/USER.md)"
```

### 3.3 Verify Mnemosyne received the memories

```bash
docker exec hermes_test-restaurant hermes memory list
```

Expected: brand facts and owner profile listed in Mnemosyne.

### 3.4 Verify Mnemosyne SQLite DB location

```bash
docker exec hermes_test-restaurant find / -name "mnemosyne.db" 2>/dev/null
```

**Expected:** DB inside `/opt/data/` — ideally:
```
/opt/data/plugins/mnemosyne/mnemosyne.db
```

Which maps to this path on the host:
```
/opt/aaas/tenants/test-restaurant/plugins/mnemosyne/mnemosyne.db
```

**If DB is inside `/opt/data/`** — ✅ persisted on host automatically via volume mount.

**If DB is OUTSIDE `/opt/data/`** — ⚠️ add a third volume mount:
```yaml
volumes:
  - /opt/aaas/tenants/test-restaurant:/opt/data
  - /opt/aaas/tenants/test-restaurant/files:/home/hermes/files
  - /opt/aaas/tenants/test-restaurant/mnemosyne.db:{actual-db-path}
```

Update `onboard-tenant.md` in Plan A if third mount is needed.

## Known Limitations (POC)

- No automated health check scheduling — run manually
- Logo must be manually added to files/assets/ post-onboarding
- No web portal — all management via OpenCode CLI
- Mnemosyne fastembed model downloads on first use (~500MB per container)
- Machine must stay on — containers stop if Windows sleeps

## Upgrade Path to v1 Production

| POC | v1 Production |
|---|---|
| Docker on WSL2 | Docker on Linux VPS (Hetzner/DigitalOcean) |
| Manual logo upload | Upload via Telegram directly |
| No backup | GitHub for config + Cloudflare R2 for files |
| Single operator via CLI | Admin web form for non-technical staff |
| Manual BotFather setup | Semi-automated via Telegram API |
| No health check scheduling | Cron-based automated monitoring |
